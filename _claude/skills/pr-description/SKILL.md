# PR description の書き方

## 最上位の原則

**書く意思があるときだけ書く**。各セクションについて「自分はこれを書きたい理由を説明できるか」を自問し、答えが「フォーマットの空白を埋めるため」「調査結果を捨てるのが惜しいから」になるなら**書かない**。空または最小限のまま残す。

受動的な情報埋めはノイズ。具体例：

- grep 残存ゼロ確認結果
- `php -l` / 構文チェックが OK だった旨
- Plan / 調査ノートの転記
- リスクの先回り説明（「もし○○すると××になる」）
- コード差分・コミットメッセージで読める情報の繰り返し

PR description は**目次**であって**営業資料**ではない。詳細はコード差分とコミットメッセージに役割分担する。

## テンプレートの HTML コメントの解釈

`<!-- 不要であれば削除 -->`、`<!-- なければ特になしでよい -->` などのコメントは「**空でもよい**」のサインであり、「**空が望ましい**」ではない。空にするのも書くのも能動的な選択になる。

## 作成前の手順

1. チームの平均的な粒度・分量を把握する：
   ```
   gh pr list --limit 20 --state merged --json title,body
   ```
   3〜5 本サンプリングして、description の長さ・章立て・固有名詞の説明粒度を確認する。

2. PR テンプレート（`.github/PULL_REQUEST_TEMPLATE.md` など）を読み、各セクションの意図を確認する。

## 各セクションの自問リスト

| セクション | 自問 |
|---|---|
| Why / 目的・背景 | レビュアーが**前提として欠かせない**情報か？ Slack やチケットで読める内容を繰り返していないか？ |
| What / 変更概要 | ファイル名の羅列になっていないか？ 層・カテゴリで括れないか？ |
| 重点レビュー | 「ここは普通のレビューで見落とされたら困る」と本気で思うか？ |
| 影響範囲（自由記述） | テンプレのリンク以外に、レビュアーに能動的に伝えたい注意点があるか？ |
| 確認方法 | レビュアーや QA に「これだけは触ってほしい」という要請があるか？ |

NO なら空または「特になし」のまま。

## 書き方のコツ

- **固有名詞のコンテキスト**: 社内ツール・社内サービス・社内略語は最初の登場で 1 行のコンテキストを添える。
  - 例: FooBar（モバイルプッシュ通知配送サービス）の利用を...
- **What の括り方**: ファイル名・関数名の羅列にせず、層・カテゴリで括る。
  - 悪い例: `XxxxxxxxxYyyyyy.php` の `functionHogeHoge` を削除…
  - 良い例: ライブラリ（PHP）／呼び出し側（コントローラ）／フロント（JS とテンプレ）
- **Why のスコープ**: 「なぜこの作業が必要か」を 1-2 文で。リスクシナリオの先回り説明は書かない（質問が出てから答える）。
- **確認方法**: 自動テスト + 主要ハッピーパス 2-3 件に絞る。DevTools の細かい観察項目などは過剰。

## アンチパターン（自分がやりがちな失敗）

- Plan モードで集めた詳細な調査ノートをそのまま description に流し込む
- 「重点レビュー」に**一般的なレビューで気づける**範囲の判断根拠を書く
- 「影響範囲調査」に grep / 静的解析 / 構文チェックの実施結果を箇条書きにする
- 確認方法のチェックボックスを 8 個以上並べる
