歴史的経緯の調査手順

# 目的

あるコード/設定がなぜ存在するのか、その歴史的経緯と現在の必要性を調査する。
当該コードが最初に追加されたのはいつか、途中で変更・削除されたのはいつかを時系列にまとめる。
また、pull requestの概要や関連するチケットの情報から **なぜその変更が行われたか** の
意味的背景をまとめ、時系列とともに提示する。

# 必要なツール/権限

- git（ローカルリポジトリ）
- gh CLI（GitHub認証済み）

## あると望ましい

- Atlassian MCP（Jira/Confluence連携）

### 実行手順

1. 現在のブランチの変更内容の確認

    git diff develop...HEAD

2. git blame で追加元コミットを特定

    git blame develop -- <file_path>

3. コミットからPRを検索

    gh pr list --search "<commit_hash>" --state all --repo <org>/<repo> --json number,title,url

4. PRの詳細を確認

    gh pr view <PR番号> --repo <org>/<repo> --json title,body,author,createdAt,mergedAt

5. **リファクタリングPRの場合、checkout して元ファイルの状態を確認**

    PRがリファクタリング（コードの切り出し・移動）の場合、`git log -S` だけでは不十分。
    リファクタリング前のコミットをcheckoutし、元ファイルをblameして真の追加地点を特定する。

    ```bash
    # リファクタリングPRの1つ前のコミットをcheckout
    git checkout <リファクタリングコミット>^

    # 元ファイルをblame
    git blame <元ファイルパス> | grep "<検索文字列>"

    # 元に戻る
    git checkout -
    ```

6. Jiraチケットで背景を確認（Atlassian MCP使用）

    mcp__atlassian__search: query="<チケット番号>"

7. **消費側コードの追跡（重要）**

    「設定側」のコードだけでなく、「消費側」のコードも追跡する。
    消費側が削除されて設定側だけ残る「片付け漏れ」パターンがある。

    ```bash
    # 設定された値がどこで使われているかを検索
    git log --all --oneline -S "<変数名や値>" -- '*.php'

    # 消費側コードが削除されたPRを特定
    gh pr diff <PR番号> --repo <org>/<repo> | grep -C 16 "<キーワード>"
    ```

8. コードの変遷を追跡（削除された経緯）

    gh pr diff <PR番号> --repo <org>/<repo> | grep -C 16 "<キーワード>"

9. **障害が関係する場合、一時対応と恒久対策の両方を追跡する（レポート作成前に必須）**

    Revert/hotfixは**一時対応**であり、その後に**恒久対策**があるはず。
    一時対応だけ見て「削除できない」と結論づけない。
    **恒久対策PRには障害の原因と対策の詳細が書かれていることが多い。レポート作成前に必ず確認する。**

    ```bash
    # Revert後にマージされた関連PRを時系列で確認
    gh pr list --search "<関連キーワード>" --state merged --repo <org>/<repo> \
      --json number,title,mergedAt | jq 'sort_by(.mergedAt)'
    ```

10. **障害発生時のコードを直接確認する（重要）**

    「現在のコード」ではなく「障害発生時のコード」を確認する。
    現在のコードは恒久対策後の状態である可能性があり、それを過去の障害に当てはめると誤った理解になる。
    特に**エラーハンドリング（例外 vs ログ出力）の違い**に注意。

    ```bash
    # 削除コミットの直前の状態を確認
    git show <削除コミット>^:<ファイルパス> | grep -A 20 "<関連メソッド名>"

    # 関連メソッドの変遷も確認（フィールドだけでなく、それを使用するメソッドも追跡）
    git log --all --oneline -S "<関連メソッド名>" -- '<ファイルパス>'
    ```

11. **現在のコードと障害原因コードを比較する（重要）**

    障害の原因となったコードが、現在のdevelop/masterでどうなっているか確認する。
    **差分があれば、既に恒久対策が入っている可能性が高い。**

    ```bash
    # 現在のdevelopでの実装を確認
    git show develop:<ファイルパス> | grep -A 20 "<問題のメソッド名>"

    # または git blame で最新の変更者/コミットを確認
    git blame develop -- <ファイルパス> | grep -A 10 "<問題の箇所>"
    ```

    障害レポートで「ここで例外が発生した」と言われている箇所が、
    現在のコードでは違う実装になっていれば、恒久対策PRを探す。

12. 以上を、変更の意味的背景がまとまるまで繰り返す

### 注意事項：よくある落とし穴

1. **同名の別概念を混同しない**
   - 例: DBカラム `table.column` と、別システムへ渡すデータ `custom_field.column` は別物
   - 検索結果に複数の文脈が混在する場合、今回の対象がどれかを慎重に判断する

2. **最も古いコミットが「起源」とは限らない**
   - 検索結果の中から、対象の文脈（例: 特定のクラスへの設定）に合致するものを選ぶ
   - 関連しそうなコミットメッセージを見逃さない

3. **「なぜ存在したか」は「どこで使われていたか」から分かる**
   - 設定側のコードだけを見ても、本当の目的は分からない
   - 消費側のコードとそのコメントを確認する

4. **「片付け漏れ」パターンを疑う**
   - 消費側が削除されたのに設定側が残っている場合、意図的な設計ではなく片付け漏れの可能性が高い

5. **Revertを見て「削除不可」と早合点しない**
   - Revertは一時対応。その後に恒久対策PRがマージされている可能性がある
   - 「過去に障害が起きた → だから削除できない」ではなく、「恒久対策は何か？」を確認する

6. **「現在のコード」を必ず確認する**
   - 過去の経緯だけでなく、現在のdevelop/masterの実装を確認する
   - 障害原因のコードが現在どうなっているか比較すれば、恒久対策の有無は明白
   - **調査の最初に行った `git blame develop` の結果を、障害原因コードと照合する**

7. **「現在のコード」を過去に当てはめない**
   - 現在のコードを読んで理解を形成し、それを過去の障害に当てはめるのは危険
   - **障害発生時のコードを直接確認する**（`git show <コミット>^:<ファイル>`）
   - 特にエラーハンドリングは恒久対策で変更されていることが多い

8. **関連するメソッド・コードの変遷も追う**
   - フィールドの変遷だけでなく、**それを使用するメソッドの変遷**も追跡する
   - フィールド削除が問題になるのは「読み書きする箇所」が原因であり、そのコードを追わないと真の原因は分からない

9. **「エラー」と「例外」を区別する**
   - **エラーログ**: 処理は続行される（深刻度は状況による）
   - **例外**: 処理が停止する（キャッチされなければ）
   - PRやチケットに「エラーを観測」と書かれていても、それが「ログ」なのか「例外」なのかをコードで確認する
